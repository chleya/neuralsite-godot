## 企划书（最终整合版）

**编制单位：NeuralSite研发团队**
**版本：V5.0（整合版）**

---

## 一、项目背景与愿景

### 1.1 行业痛点

当前道路工程施工管理面临诸多挑战：

- **信息孤岛**：图纸、现场、方案、人员、资金、进度分散在不同系统，缺乏统一数据底座
- **BIM困境**：传统BIM系统沦为「观摩展示」，实际施工指导价值有限
- **数据断层**：各方数据格式不统一，设计院、施工单位、监理、业主之间流转困难
- **网络困境**：山区、隧道等偏远地区网络信号差，信息系统形同虚设
- **录入负担**：一线人员本已繁忙，数据填报成为「额外负担」，缺乏动力

### 1.2 核心愿景

打造一个「数据驱动、语义智能、四维可视」的智慧工程管理平台：

- 以「毫米级空间+标签」为核心数据架构
- 将工程实体、施工状态、资源流转统一管理
- 通过语义标签系统实现智能分类和检索
- 依托四维可视化技术直观呈现施工进度
- 实现AI辅助决策和自动化施工指挥

### 1.3 技术路线演进

项目经历了三次重大技术演进：

| 阶段 | 特征 |
|------|------|
| V1.0 | 传统工程管理到智慧工地 |
| V2.0-V3.0 | BIM数字孪生到认知孪生 |
| V4.0-V5.0 | 功能堆砌到数据驱动 |

**核心设计理念**：「代码即配置、数据即资产、语义即智能」

### 1.4 与BIM的本质区别

| 维度 | 传统BIM | NeuralSite |
|------|---------|------------|
| 用户定位 | 设计师/工程师 | 施工队长/班组长/一线工人 |
| 核心场景 | 办公室设计分析 | 工地现场管理决策 |
| 数据流向 | 设计→施工→运维 | 施工→管理→决策 |
| 价值交付 | 可视化模型 | 可操作数据 |

## 二、系统架构设计

### 2.1 整体架构

系统采用「后端核心+可视化前端+数据接口」三层架构：

```
┌─────────────────────────────────────────────────────────────┐
│                    可视化前端层 (Godot 4.3)                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │ 三维场景模块 │  │ 时间轴模块  │  │  交互模块   │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
├─────────────────────────────────────────────────────────────┤
│                      API接口层 (REST + WebSocket)            │
├─────────────────────────────────────────────────────────────┤
│                    后端核心层 (FastAPI)                      │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │ 空间引擎子系统│  │ 实体管理子系统│  │ 状态追踪子系统│          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
│  ┌─────────────┐  ┌─────────────┐                          │
│  │ 资源关联子系统│  │ 智能服务子系统│                          │
│  └─────────────┘  └─────────────┘                          │
├─────────────────────────────────────────────────────────────┤
│                      数据层 (PostgreSQL + SQLite)            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心设计理念

**理念一：离线优先——让网络不再是问题**

- 网络不是前提，而是补充；终端不是附庸，而是主体
- 终端即数据库，离线时正常操作，联网后自动同步
- 采用CRDT算法处理多人离线操作后的数据合并

**理念二：极简交互——让操作像呼吸一样自然**

- 「一个界面，一个动作」，把复杂流程拆解为简单的「点击—完成」
- 默认智能推荐、拍照为主文字为辅、手势化操作
- 整改照片填报：从传统的2-3分钟缩短到10秒

**理念三：游戏激励——让认真的人被看见**

- 经验值代表能力成长、金币代表物质回报、荣誉值代表行业认可
- 即时的、正向的反馈机制让施工人员从「被动填报」变为「主动参与」

**理念四：隐形WBS——让结构成为底层而非前台**

- WBS作为底层数据结构而非前台交互元素
- 用户感受不到WBS存在，但系统在后台支撑一切

## 三、核心数据模型

### 3.1 毫米级空间+标签架构

**核心突破：从「存点」到「存关系」**

传统方案存每个点：1公里道路 × 毫米精度 = 10亿个点 → 数据爆炸

新方案存关系：
```
空间 = 无限延伸的连续坐标（桩号+横距+高程）
    ↓
实体 = 贴在空间上的「贴纸」（路基、桥梁、涵洞...）
    ↓
标签 = 贴在实体上的「属性」（类型、状态、材质...）
    ↓
过程 = 贴纸随时间变化的「快照」
```

| 方案 | 1公里道路数据量 |
|------|----------------|
| 传统（存每个点） | 10亿个点 |
| 新方案（存实体+标签） | ~5000实体 + ~10万条过程数据 |

### 3.2 四类核心数据模型

**第一类：空间定位模型**

采用「桩号+横距+高程」三维坐标体系：
- 桩号：公路标准K0+000格式，支持毫米级精度
- 横距：相对线路中线的横向偏移（正值=左侧，负值=右侧）
- 高程：绝对标高

空间计算引擎支持：
- 直线段的线性插值
- 圆曲线的弧长和切线角
- 缓和曲线的回旋线积分
- 纵曲线参数自动计算设计高程

**第二类：工程实体模型**

实体叠加在空间上，代表各种工程对象：
- 路基、桥梁、涵洞、隧道、边坡、排水设施、路面、基础

属性：
- 几何属性：起始桩号、终止桩号、横向偏移、基础高程、宽度、高度
- 业务属性：材质类型、施工单位、验收状态、设计参数

**第三类：状态快照模型**

追踪施工进度和质量状态：
- 时间戳、状态类型（规划→清表→土方→路面→收尾→完成）
- 进度百分比、质量状态（合格/不合格/待检）
- 现场照片、工程量数据、区块链存证哈希

**第四类：资源关联模型**

人员、资金、设备、材料通过关系网络与实体关联：
- 关系类型：人员分配、资金拨付、设备使用、材料消耗
- 支持多对多关联，便于资源调配分析和成本归集

### 3.3 数据库分层设计

```
┌─────────────────────────────────────────────────────────────┐
│                     主业务层                                 │
│  work_areas | tasks | issues | daily_reports               │
│  engineering_quantities | design_quantities                │
│  resource_logs | design_spatial_objects                     │
│  terrain_raw_objects | terrain_change_sets                   │
├─────────────────────────────────────────────────────────────┤
│                     空间层                                   │
│  spatial_raw_objects | spatial_bindings                     │
│  spatial_display_objects                                    │
├─────────────────────────────────────────────────────────────┤
│                     历史层                                   │
│  task_status_history | issue_status_history                │
│  work_area_progress_history | quantity_progress_history      │
└─────────────────────────────────────────────────────────────┘
```

## 四、功能模块详细设计

### 4.1 空间计算引擎

**功能**：
- 桩号↔三维坐标双向转换
- 平曲线/圆曲线/缓和曲线/纵曲线计算
- 横断面生成
- LRU坐标缓存（默认10000条）

**API**：
- `POST /v1/space/station-to-coord` - 桩号转坐标
- `POST /v1/space/coord-to-station` - 坐标转桩号
- `GET /v1/space/nearby` - 邻近桩号列表
- `GET /v1/space/distance` - 两桩号间距离

### 4.2 实体生命周期管理

**功能**：
- CRUD操作，多维度查询
- 父子/邻接/依赖关系管理
- 变更历史记录

**API**：
- `GET/POST /v1/entities` - 列表/创建
- `GET/PUT/DELETE /v1/entities/{id}` - 详情/更新/删除
- `GET /v1/entities/at-location` - 按桩号查询

### 4.3 状态版本系统

**五种版本类型**：
| 类型 | 说明 |
|------|------|
| BASELINE | 设计完成时的规划状态 |
| CURRENT | 实时实际施工状态 |
| PLANNED | 排产计划中的预期状态 |
| SIMULATED | 假设情景分析 |
| FORECAST | 历史趋势外推的未来状态 |

**核心能力**：
- 时间旅行查询：输入目标时间，返回该时间点之前最近的状态快照
- 进度计算器：单体进度、区段进度、完成预测

**API**：
- `GET /v1/states/entity/{entity_id}/history` - 历史状态
- `POST /v1/states` - 创建快照
- `GET /v1/states/latest` - 最新状态

### 4.4 语义标签体系

**九大标签分类**：

| 分类 | 说明 |
|------|------|
| construction | 施工状态 |
| quality | 质量等级 |
| material | 材料类型 |
| safety | 安全等级 |
| cost | 成本分类 |
| progress | 进度状态 |
| inspection | 验收状态 |
| risk | 风险等级 |
| custom | 自定义 |

**标签属性**：唯一标识、分类、名称、值、描述、颜色、优先级、元数据

**API**：
- `GET /v1/semantic/tags/definitions` - 所有标签定义
- `POST /v1/semantic/tags/apply` - 应用标签
- `GET /v1/semantic/tags/entity/{entity_id}` - 实体标签

### 4.5 资源关联网络

**关联类型**：
- 人员关联：工种、班组、资质
- 资金关联：预算类别、拨付金额、使用状态
- 设备关联：设备类型、使用时长
- 材料关联：材料类型、消耗量

**API**：
- `GET /v1/funds/statistics/summary` - 资金统计
- `GET /v1/personnel/attendance` - 人员考勤

### 4.6 四维可视化系统 (Godot)

**核心模块**：
- 三维场景模块：道路模型、桥梁结构、地形地貌实时渲染
- 时间轴模块：播放/暂停/拖动/倍速控制
- 交互模块：点击选中、信息弹窗、视角切换

**施工阶段颜色编码**：

| 阶段 | 颜色 |
|------|------|
| planning | 蓝色半透明 |
| clearing | 橙色 |
| earthwork | 棕色 |
| pavement | 灰色 |
| finishing | 浅绿色 |
| completed | 深灰色 |

## 五、技术规格

### 5.1 后端技术栈

| 组件 | 技术 |
|------|------|
| 框架 | FastAPI (Python 3.10+) |
| 数据库 | PostgreSQL + PostGIS / SQLite |
| 缓存 | Redis |
| 任务队列 | Celery |
| API文档 | /docs (Swagger), /redoc |

### 5.2 前端技术栈

| 组件 | 技术 |
|------|------|
| Web前端 | React + Three.js + ECharts |
| 移动端 | Flutter |
| 3D引擎 | Godot 4.3 |

### 5.3 数据格式规范

**桩号格式**：`K里程+米.毫米`（如K1+500.500）

**坐标格式**：
```json
{
  "x": 123456.789012,
  "y": 987654.321012,
  "z": 100.0000
}
```

**时间格式**：ISO 8601（如`2026-03-25T13:01:42Z`）

**GeoJSON**：用于Godot端数据交换
